## 1. 용어

| 용어 | 설명 |
|------|------|
| **t0 / t_end** | 슬라이딩 윈도우에서 과거 실측 구간의 마지막 시각 (`t_end = start + L - 1`). wide fan은 보통 이 행에서 읽는다. |
| **lead** | 예보 기준 시각으로부터의 전방 시간 인덱스 (`1…H_wide`). |
| **issue_time** | 예보 발표 시각(누수 규칙과 함께 사용). |
| **target_time** | 예보값이 의미하는 검증 시각 τ. |
| **wide fan** | 한 parquet 행에 `fcst_{var}_{h:03d}` 형태로 펼쳐진 다중 lead 예보 열 집합. |
| **per-target (narrow)** | 각 행이 하나의 τ에 대응하고 lead 축을 열로 펼치지 않은 스키마(방식 B). |

## 2. 방식 A (현재 Track B wide 마트) 개요

Track B per-site mart는 `feature_mart_track_b_per_site` 등에 저장된다. `dataset/preprocessor/track_b_forecast_join.py`, `track_b_enrich_mart.py`가 `era5_hourly_valid` / `issue_target` 등 조인 모드에 따라 `fcst_*` 열을 채운다. 학습 코드는 기본적으로 과거 `FEATURE_COLS`만 텐서로 올리고, **`fcst_*`는 파일에만 존재**할 수 있다(플래그로 통합 입력 시 소비).

### 2.1 스키마 요약

- **행**: 시간 인덱스(1시간). **열**: 관측·파생 + `fcst_*` wide fan.  
- **의미**: 한 시각 `t0`에서 본 미래 `H_wide`시간까지의 예보를 같은 행에 펼친다.

### 2.5.1 Parquet 한 행(저장 단위)

- 배치가 아니라 **저장 단위** 한 행에 실측 채널 + `fcst_*` fan 열이 함께 있다.

### 2.5.2 현재 파이프라인 (`pv_dataset` + `train_tslib_model`) — 단일 `x`

| 항목 | Shape (단일 샘플) | 설명 |
| --- | --- | --- |
| `x` | `[L, C_obs]` | 인덱스 `start … start+L-1`의 **연속 L시간**, 채널은 `FEATURE_COLS`만 (`C_obs=32`). |
| `y` | `[H_pred, C_obs]` | 인덱스 `start+L … start+L+H_pred-1`의 **미래 H_pred시간**, 동일 32채널 전부. |


`DataLoader`가 배치로 묶으면:


| 텐서  | 배치 shape             | 비고                                  |
| --- | -------------------- | ----------------------------------- |
| `x` | `[B, L, C_obs]`      | 예: `[32, 168, 32]`                  |
| `y` | `[B, H_pred, C_obs]` | 손실에서 보통 `normalized_power` 슬라이스만 사용 |



`train_tslib_model.py`에서 TSLib 호출은 `model(x, None, None, None)` 형태로, **입력은 위 `x` 하나**가 기본이다(`x` = `x_enc`).

이 경로에서는 **mart에 `fcst_*`가 있어도 로드하지 않으므로**, wide는 **디스크에만 존재**하고 학습 텐서에는 **들어가지 않는다**.

---

#### 2.5.3 방식 A를 쓸 때 — **구현별** 학습 배치 shape (Track B wide 소비)

마트는 wide 그대로 두고, **Dataset에서만** 아래처럼 조합하면 된다. (`t_end = start + L - 1`)

**패턴 A-1 (권장 개념): 과거 L창 실측 + “마지막 시각”에서 fan만**


| 출력       | shape                                    | 내용                                                                                                |
| -------- | ---------------------------------------- | ------------------------------------------------------------------------------------------------- |
| `x_hist` | `[B, L, C_obs]`                          | 기존과 동일: `data[start:start+L, FEATURE_COLS]`                                                       |
| `x_fcst` | `[B, H_pred, V]`                         | **행 `t_end` 한 줄**에서 `fcst_{var}_{h}`를 `(h, var)` 순으로 채움. `H_pred ≤ H_wide`이면 lead `1…H_pred`만 사용. |
| `y`      | `[B, H_pred, 1]` 또는 `[B, H_pred, C_obs]` | 미래 구간 `start+L …`의 실측(전력 등). 기존과 동일하게 잡을 수 있음.                                                    |


`DataLoader`의 `collate_fn`에서 튜플 `(x_hist, x_fcst, y)`를 반환하면, **모델은 두 입력**을 받도록 바꿔야 한다(예: `torch.cat`으로 채널 merge, 또는 별도 브랜치).

**패턴 A-1d (권장·단일 텐서)**: §**2.6**과 같이 `[B, L+H, N_feature]` 로 묶어 `x_enc` 하나로 넘긴다. 채널 정렬·마스크·`seq_len=T`는 §2.6을 따른다. **구현**: `train_tslib_model.py`의 `--merge-future-nwp-into-encoder-input`(및 `--future-nwp-variable-names`).

**패턴 A-1b: fan을 벡터로 펼쳐 고정 길이 부가 특성**


| 출력            | shape                                   | 내용                                                                           |
| ------------- | --------------------------------------- | ---------------------------------------------------------------------------- |
| `x_hist`      | `[B, L, C_obs]`                         | 동일                                                                           |
| `x_fcst_flat` | `[B, H_pred × V]` 또는 `[B, 1, H_pred×V]` | `t_end` 행의 fan을 행 우선 또는 `(var,h)` 순으로 flatten. **시간축 1토큰**에 미래 예보 전체를 박는 형태. |
| `y`           | `[B, H_pred, 1]`                        | 동일                                                                           |


**패턴 A-1c: encoder 입력 채널에 fan을 concat (단일 `x`)**

한 시점의 입력 채널을 `C_obs + V`로 두고, **168시간 동안 fan을 반복하거나**, 마지막 시점만 fan 채널이 비영·앞은 0 등 **설계가 필요**해 비교적 덜 깔끔하다. shape 예:

- `x_merged`: `[B, L, C_obs + V_fan]` — `V_fan`은 `H_pred×V`를 채널로 쪼개거나 요약한 차원.

**패턴 A-2 (비권장): 매 시각마다 wide fan 전부를 encoder에**

- 예: 각 `t`에 대해 `fcst_*` 전 열을 읽어 `x`가 `[B, L, C_obs + V×H_wide]`에 가까워짐. **중복·해석 혼란**이 커서 문서상 비권장.

#### 2.5.4 TSLib(`train_tslib_model`)과의 정합

- 현재 스크립트는 **`x`만** 넘긴다 → **패턴 A-1**(이중 텐서)을 그대로 쓰려면 `forward`·학습 루프 수정이 필요하다.
- **패턴 A-1d / §2.6**처럼 `**[B, L+H, N_feature]` 단일 텐서**로 만들면, **기존 `model(x, None, None, None)` 호출 형태를 유지**할 수 있다(단, `seq_len`·`enc_in`을 §2.6.5에 맞게 바꿀 것).
- `enc_in` / `dec_in` 채널 수는 **실제로 `model(...)`에 들어가는 마지막 차원**과 일치해야 한다.

#### 2.5.5 숫자 예시 (대표값)

`L=168`, `H_pred=24`, `C_obs=32`, `V=9`, `H_wide=72`일 때:

- **현재(Track B 미사용)**: `x`는 `[B,168,32]`, `y`는 `[B,24,32]`.
- **A-1**: `x_hist` `[B,168,32]`, `x_fcst` `[B,24,9]` (마지막 행에서 lead 1…24), `y` `[B,24,1]` 등.
- **A-1b**: `x_fcst_flat`은 `[B, 216]` (= 24×9).
- **A-1d (통합 텐서, 아래 §2.6)**: `x_unified` `[B, 192, N]` (= 168+24), `y` `[B,24,1]` 등.

### 2.6 통합 입력 `**[B, seq_len + pred_len, N_feature]`** (권장 방향 — 단일 `x_enc`)

**목표**: TSLib 등이 기대하는 **단일 시계열 텐서** `x_enc`로 넣되, 시간축 앞쪽은 **과거 실측 구간**, 뒤쪽은 **미래 예보(또는 wide fan에서 펼친 lead 구간)** 를 같은 채널 축에서 **물리 변수 기준으로 정렬**한다.

#### 2.6.1 Shape


| 기호          | 의미                         |
| ----------- | -------------------------- |
| `L`         | 과거 실측 윈도 길이 (예: 168)       |
| `H`         | 예측 지평 (예: 24 또는 72)        |
| `T = L + H` | 통합 시계열 길이 (예: 240)         |
| `N_feature` | 아래 **채널 스키마**로 고정한 통합 채널 수 |


**배치 shape**: `x_unified`는 `[B, T, N_feature]`.  
타깃 `y`는 기존과 같이 `[B, H, …]` (미래 `H`스텝의 전력 등)로 두고, 손실은 **미래 구간 인덱스**에만 걸면 된다.

#### 2.6.2 채널 정렬 규칙 (실측 vs 예보 개수가 달라도 OK)

**근거 코드·설정**: ASOS 원천→mart 컬럼은 `dataset/preprocessor/config.py`의 `ASOS_FEATURE_COLS`; mart에 붙는 ERA5 격자 컬럼은 `ERA5_FEATURE_COLS`; Track B `fcst_*`의 값 열 이름(단기와 동일 슬롄)은 `TRACK_B_SERVICE_FCST_COLS` / `track_b_build_report.json`의 `fcst_value_cols`. `join_mode=era5_hourly_valid` 이고 `fcst_schema=shortterm_aligned`일 때 ERA5→단기 이름 유도는 `dataset/preprocessor/track_b_forecast_join.py`의 `era5_hourly_to_shortterm_service_frame`과 동일 규칙이다.

아래 표는 **물리량 단위로 같은 통합 채널 인덱스**에 두기 위한 **권장 매핑**이다. (`fcst_schema=era5_native`이면 `fcst_*` 열 접두가 `t2m_c` 등으로 달라지므로 **별도 표**를 둘 것.)

##### (2) 예보·ERA5 유도에만 있고 ASOS에 직접 없는 슬롄 — **본 저장소 학습에서의 확정**

- `pty`, `pop`, `sno`는 **통합 인코더 입력에서 제외**한다(실측 mart에 대응 열이 없고, ERA5→단기 유도값도 품질 검증이 필요). `--future-nwp-variable-names` 기본값에서 빠져 있으며, fan에서도 읽지 않는다.
- **실측에만 있는 채널**(`pa`, `si`, `ss`, 파생·캘린더 등)은 **통합 `FEATURE_COLS`에 그대로 포함**한다. 미래 `H`행에서는 값을 두지 않거나 0으로 두고, **마지막 채널의 구간 플래그**(과거=1 / 미래 NWP 구간=0)로 구간을 구분한다(`pv_dataset` 구현).
- 마트 빌드 단계에서 `fcst_*`를 줄이려면 여전히 `dataset/preprocessor/run.py`의 `--fcst-value-cols`로 조정할 수 있다.

##### (3) mart 실측 행에만 있고 `fcst_*`(단기 9종)에는 없는 대표 항목


| 항목        | ASOS / mart                               | ERA5 mart | 단기 `fcst_*`                         | 비고                                            |
| --------- | ----------------------------------------- | --------- | ----------------------------------- | --------------------------------------------- |
| 기압        | `pa`                                      | —         | 없음                                  | 미래 행은 패딩+마스크.                                 |
| 일사·대체 일사  | `si`                                      | —         | 없음 (단기예보에 일사 변수 없음, README 전략 B 참고) |                                               |
| 일조        | `ss`                                      | —         | 없음                                  |                                               |
| 전력·파생·캘린더 | `normalized_power`, `pv_roll_*`, `hour` 등 | —         | 없음                                  | 통합 시 **전력은 보통 `y`로 분리**하고 입력에서는 마스킹하거나 별도 정책. |


##### (4) 번호 규칙 (기존 2.6.2) — 표와 함께 유지

1. **공통 기상·공변량**: 위 표의 **동일 물리량**은 통합 텐서에서 **동일 채널 인덱스** `k`에 둔다. (이름이 `hm` vs `reh`처럼 달라도 **슬롄 정의는 하나**로 고정.)
2. **실측에만 있는 채널**: 예보 구간 행 `t = L … T-1`에서는 패딩 + **마스크 무효**.
3. **예보(또는 ERA5 유도 단기 슬롄)에만 있는 채널**: 과거 구간 `t = 0 … L-1`에서 패딩 + 마스크. **표 (2)처럼 ASOS에 없는 슬롄을 아예 빼고 `fcst_value_cols`를 줄이는지** 빌드·Dataset 계약으로 확정할 것.
4. **`N_feature`**: 현재 구현은 `len(FEATURE_COLS) + 1`(구간 플래그 1채널). `pty`/`pop`/`sno`를 제외한 만큼 fan 열만 요구한다.

이렇게 하면 **모델·스크립트는 채널 인덱스만 보면** “이 슬롄이 기온이다”가 과거·미래 구간에서 **일치**한다.

#### 2.6.3 마스킹 (0 패딩만으로는 부족할 수 있음)

- **0 또는 `nan`만** 두면, DLinear 등 **값 0을 유의미한 신호로 학습**할 위험이 있다.
- **권장**: 최소 **구간 마스크** 1채널(`t < L`이면 1, 아니면 0) 또는 **채널별 유효 마스크**(실측/예보/둘 다 가능 구간)를 `N_feature`에 **추가 채널**로 넣는다.
- 손실은 **미래 전력 타깃**에만 두되, 필요하면 **마스크된 위치의 보조 손실**은 쓰지 않는다.

#### 2.6.4 행 `t`의 의미 (방식 A vs B)


| 출처                 | 행 `0 … L-1`               | 행 `L … T-1`                                                                                                                                                |
| ------------------ | ------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **방식 B + 연속 mart** | 실제 시계열의 연속 **실측 시각** `H`행 | 연속 **미래 시각** `τ`마다 **그 시각 예보** 1벌                                                                                                                          |
| **방식 A (wide)**    | 동일: 연속 실측 행               | **parquet의 연속 행이 아닐 수 있음**. 보통 **마지막 실측 시각 `t_end` 한 행의 fan**에서 lead `1…H`를 꺼내 **가상의 `H`개 시간스텝**으로 쌓아, 행 `L+k`에 lead `k+1` 예보 벡터를 둔다. (채널 정렬은 §2.6.2와 동일.) |


#### 2.6.5 모델·설정(`seq_len`, `enc_in`)과의 정합

- TSLib 계열은 `x_enc`의 시간 길이가 **`configs.seq_len`과 일치**하는 전제로 동작하는 경우가 많다. 통합 모드에서는 **`seq_len := T = L + H`**, **`enc_in := len(FEATURE_COLS)+1`** 로 `train_tslib_model.py`가 설정한다.
- **`pred_len`**: 기존과 같이 `H`로 두면, 헤드 출력 `[B, H, c_out]`과 `y`의 시간 길이가 맞는다(프로젝트의 `train_tslib_model` 손실 슬라이스와 동일 패턴).
- **주의 (아키텍처 해석)**: PatchTST 등 **시간축 양방향 self-attention**을 쓰면, 과거 토큰이 미래(예보) 토큰을 **볼 수 있다**. 이는 “**미래 기상 예보를 이미 알고 전력을 맞추는**” 설정으로는 **합리적**이나, “**과거 실측만으로 미래를 예측**”하는 순수 인과 설정과는 다르다. 의도한 운영 시맨틱과 맞는지 팀에서 고정할 것. (인과만 허용하려면 마스킹된 attention·별도 브랜치·과거만 넣는 인코더 등이 필요하다.)

#### 2.6.6 타당성 요약


| 판단             | 내용                                                                                   |
| -------------- | ------------------------------------------------------------------------------------ |
| **가능·합리**      | 단일 `[B,T,N]`으로 **기존 `model(x,…)` 경로**와 맞추기 쉽고, 채널 정렬·패딩·마스크 규칙만 문서·코드에 고정하면 재현성이 좋다. |
| **반드시 문서화/구현** | 통합 채널 순서표, 마스크 채널, `seq_len=T`, 방식 A에서 미래 `H`행을 fan에서 어떻게 쌓는지.                       |
| **검토 필요**      | 양방향 모델에서의 **정보 흐름(미래 예보 가시성)** 이 비즈니스 요구와 일치하는지.                                     |


#### 2.6.7 구현 연동 (`pv_dataset` + `train_tslib_model`)

- **기본(플래그 없음)**: `x`는 `[L, 32]`, `y`는 `[H, 32]` — 기존과 동일.
- **`--merge-future-nwp-into-encoder-input`**: `x`는 `[L+H, 33]`(32 + 구간 플래그). fan 변수는 `--future-nwp-variable-names`(기본 `tmp,reh,wsd,vec,sky,pcp`). wide `fcst_*` 마트 필요.
- **방식 A**: 미래 `H`행은 `t_end = start + L - 1` 행의 lead `1…H` fan.

## 3. 방식 B — 행 = 검증시각 τ, “그 시각에 해당하는” 예보만 (협의 스키마)

### 3.1 목표

각 행 `τ`에 대해:

- **실측·파생 feature**: 지금 mart와 동일 (그 시각 τ의 값).
- **예보 feature**: “**검증시각 τ**를 target으로 하는” 예보 1벌 (변수당 1값 또는 소수 열).

즉 **lead 축을 열로 펼치지 않고**, “**target_time = τ**”에 매칭되는 한 점만 둔다.

### 3.2 반드시 고정할 계약: “어느 issue의 예보를 τ행에 넣는가”

운영에서 흔한 규칙 예:

- **R1 (직전 이슈)**: `issue* = max{ issue_time | issue_time ≤ τ - lag_available }`, 그 issue의 `target_time==τ` 행 사용.
- **R2 (고정 스케줄)**: 매일 05:00 발표만 사용 등.
- **R3 (t0-정렬)**: 학습 샘플의 “예보 기준 시각”을 항상 **윈도우의 마지막 관측 시각**으로 두고, τ > t_end 인 미래 구간만 예보 행을 채움 (아래 4절과 결합).

방식 B 마트를 빌드하려면 **R1/R2/R3 중 하나를 문서·코드에 고정**하고, long 예보 테이블에 대해 **merge 키 `(cid_seq, target_time)` + issue 선택**으로 구현한다.

### 3.3 구현 경로

#### 경로 B-1: **전처리(빌드) 단계에서** narrow mart 생성 (권장: 재현성)

1. 입력: 기존 `feature_mart_per_site` + long `issue_time,target_time,...`.
2. 각 split parquet의 각 행 시각 `τ`에 대해:
  - `fcst_tmp@τ`, `fcst_reh@τ`, … (이름은 팀 규칙; **접미 `_001` 제거** 등) 한 벌 계산.
3. 출력: `feature_mart_track_b_narrow/` 등 별도 트리.

**장점**: 학습 시 읽기 단순, 열 수 감소.  
**단점**: 빌드 로직·용량 정책이 하나 더 필요.

#### 경로 B-2: **런타임(Dataset)**에서만 narrow로 변환

- wide mart를 읽은 뒤, `τ`마다 “어느 t0 행의 어느 lead 열이 τ를 가리키는가” 역산:
  - wide 의미가 “`fcst_{col}_h(t0)` = 값 at `t0+h`”이면, **target τ**에 대한 값은 `**t0 = τ - h` 행의 `fcst_{col}_h`**에서 온다.
  - 단, **한 τ**에 대해 후보 `(t0,h)`가 여러 개라 **issue 규칙과 함께** unique하게 고르는 절차가 필요하다.
- **장점**: 저장 1종.  
**단점**: 구현·디버깅 난이도 상승, 행마다 조회 비용.

### 3.4 방식 B의 장·단점


| 장점                                                         | 단점                                                       |
| ---------------------------------------------------------- | -------------------------------------------------------- |
| 행이 **지금의 실측 mart와 동일한 직관**(시간축만 따라가면 됨).                   | **issue 선택 규칙**을 빠뜨리면 누수·불일치 발생.                         |
| “**240 연속 행** = 앞 168 실측 + 뒤 72는 매 시각 예보만” 같은 학습 텐서와 잘 맞음. | `era5_hourly_valid` wide와 **동일 정보를 중복 표현**할 때 정의를 맞춰야 함. |

## 4. 방식 B + 연속 240 윈도우 (`seq_len=168`, `pred_len=72`)

### 4.1 사용자 제안 정리

- 인덱스 `start`부터 **연속 `seq_len + pred_len` = 240**행을 읽는다.
- **행 `start` … `start+167`**: 전력 + **실측(및 허용된) 공변량**만 채널로 사용.
- **행 `start+168` … `start+239`**: 각 시각은 **미래**이므로, **예보 feature만** 채우고 실측 전력은 타깃으로 따로 두거나 마스킹한다.

### 4.2 텐서 `[240, N_feature]` / `[B, 240, N_feature]` 로 맞추는 방법

**§2.6과 동일한 계약**을 쓰는 것이 권장된다.

- **시간축**: 연속 `240 = L + H`행을 한 덩어리로 `x_unified`에 넣는다 → `**[B, 240, N_feature]`**.
- **채널**: 실측 전용·예보 전용·공통 기상량을 **동일 변수 = 동일 채널 인덱스**에 두고, 없는 구간은 **패딩 + 마스크 채널**(§2.6.3)로 구분한다.
- **0~167행**: 해당 시각 **실측**을 채널 스키마에 맞게 채움. 예보 전용 슬롯은 패딩 + 마스크.
- **168~239행**: 해당 시각 **예보**만 채움. 실측 전용 슬롯은 패딩 + 마스크. (전력 등 학습 타깃은 **`y`**로 분리해 손실에만 사용하는 패턴이 일반적이다.)

**두 개의 텐서** `(x_hist, x_fcst)`를 유지할 수도 있으나, 단일 `x_enc` 경로와 맞추려면 위 **통합 텐서**가 §2.6과 정합된다.

### 4.3 타깃 `y` 정의

일반적으로:

- `y[k] = normalized_power` at time index `start + 168 + k` for `k=0..71`  
(미래 72시간 **실측** 전력; 학습 시에만 존재)

예보 입력은 **그 시각의 예보 feature**이고, 손실은 **동일 시각의 실측 전력**과 비교한다.

### 4.4 인덱스·누수

- 윈도우가 `train` split 안에서만 잘리도록 **기존 per-site split**과 동일하게 `indices` 범위를 잡는다.
- **미래 72행**이 `valid`/`test`로 넘어가면 안 되는지는 **split 경계** 정책에 따른다.  
  - 엄격하게 하려면: `start+239 ≤ train_end` 같은 조건.
- 방식 B narrow에서 **τ행의 예보**가 **τ 이전 정보로만** 결정되었는지 검증 (`issue_time ≤ 기준시각` 등).

### 4.5 의사코드 (`__getitem__`)

```text
def __getitem__(i):
    start = self.indices[i]
    block = self.df.iloc[start : start + 240]  # 반드시 연속 시간 인덱스 검증

    x_hist = block.iloc[0:168][OBS_COLS].to_numpy()     # [168, n_obs]
    x_fcst = block.iloc[168:240][FCST_NARROW_COLS].to_numpy()  # [72, n_fcst]

    y = block.iloc[168:240][TARGET_COL].to_numpy().reshape(72, 1)  # 또는 전 채널

    # §2.6: [240, N_feature] 통합 + 마스크 채널 후 unsqueeze(0) → 배치 [B,240,N]
    return pack_unified(x_hist, x_fcst, y)
```

## 6. 정규화·스케일

- **실측 채널**: 기존 `scaler_stats.json` (train 구간 fit) 그대로 적용 가능.
- **`fcst_*`** (wide 또는 narrow):
  - **별도 scaler**를 두거나,
  - 단기와 ERA5 proxy를 맞춘 뒤 **동일 스케일**을 쓸지 정책 결정.
- 방식 A에서 **fan만** 떼어 쓰면, scaler는 보통 **실측 채널 통계와 분리**하는 것이 안전하다.
- **§2.6 통합 `[B,T,N]`**: 동일 채널 인덱스에 **실측·예보가 번갈아** 들어오므로, **한 슬롯에 scaler를 하나** 둘지 **출처별로 둘지**(동일 변수라도 분포가 다름) 팀 규칙을 고정한다.

## 7. 구현 체크리스트

- **시맨틱 고정**: `era5_hourly_valid` vs `issue_target` vs narrow **issue 규칙**을 문서 한 줄로 고정했는가.
- **누수**: `issue_time ≤ 기준시각`, split 경계, 미래 72행이 train 밖으로 나가지 않는지.
- **정렬**: parquet가 시간순인지, DST/결측 행으로 **연속성**이 깨지지 않는지.
- **모델 I/O**: `[B, L+H, N_feature]` 통합 텐서(§2.6) + 구간 플래그, 또는 `(hist, fcst)` 분리.
- **`seq_len` / `enc_in`**: `--merge-future-nwp-into-encoder-input` 시 `seq_len=L+H`, `enc_in=len(FEATURE_COLS)+1` (`train_tslib_model.py`).
- **Attention 해석**: 미래 예보 구간을 인코더에 넣는 것이 **운영 시맨틱·누수 정책**과 맞는가(§2.6.5).
- **기존 실험과 비교**: `enc_in`, 체크포인트, 스크립트 인자(`--seq-len`, `--pred-len`) 일관성.

## 9. 요약 문장

- **현재 Track B**는 “한 시각 t0에서 본 미래 H시간 팬”을 같은 행의 열로 저장한다. 학습에서는 보통 마지막 문맥 시각의 fan 또는 별도 파생 텐서로 소비한다.  
- **행당 τ 예보만**은 issue/target 규칙을 추가로 고정한 뒤 마트를 재빌드하거나 Dataset에서 역산해야 하며, 168+72 연속 행 전략은 타깃 시각·누수·마스킹만 맞추면 구현 가능하다.  
- **단일 입력 `[B, seq_len+pred_len, N_feature]`**(§2.6)에서는 채널을 큰 스키마에 맞춘 뒤 패딩·마스크를 쓰고, 동일 기상 변수는 동일 인덱스에 둔다. TSLib 단일 `x_enc` 경로와 맞추기 쉽다. 다만 `seq_len`을 통합 길이 `L+H`로 둘지, 양방향 attention에서 미래 예보 토큰을 볼 수 있다는 점은 설계로 명시한다.
- `pv_dataset` 확장 또는 별도 Dataset + 학습 스크립트 분기로 연결하는 것이 일반적이다.
